The Shield: Threat Model

Honesty starts with admitting what didbox402 cannot protect. While we aim for maximum privacy, the physical reality of hosting and networking creates metadata "exhaust."

1. Metadata Leakage (The Social Graph)

Threat: The storage provider can see who is communicating with whom by observing IP addresses and salted DID hashes.

Mitigation: Use Salted DID Hashing for inbox IDs and encourage agents to use Tor/VPNs to mask network origin.

2. Payload Size & Frequency (Traffic Analysis)

Threat: Even if data is encrypted, the size of the ciphertext and retrieval frequency can leak the payload type.

Mitigation: Clients should implement Padding (making all blobs a fixed size or multiples of a block) to normalize traffic profiles.

3. Malicious Provider (The Data Trap)

Threat: A provider could accept payment but delete data early or deny valid retrieval requests.

Mitigation: didbox402 is a Commodity Protocol. Agents should diversify storage across multiple nodes.

4. Replay Attacks

Threat: An attacker intercepts a signed storage request and resubmits it to drain the client's funds.

Mitigation: X-DID-Signature must bind to the Request Hash (Method + Path + Body Hash) and include a nonce.

Trust Assumptions

  • Client Integrity: We assume the client handles their keys securely and encrypts data locally.
  • Infrastructure: We trust the underlying cloud (R2/S3) to maintain bit-integrity for the lease duration.
