The Hub: Multi-Inbox Organization v0.6.0

As agents scale, a single "Inbox" becomes a bottleneck. didbox402 supports Named Inboxes, allowing agents to partition their incoming data flows by project, group, or priority.

The "Edges" of Organization: Each inbox is cryptographically isolated. Knowing the address of one inbox does not reveal the others.

Cryptographic Mechanism

An inbox is defined by its Alias (a string like "project-x" or "alerts"). The server identifies an inbox using a salted identity hash:

Inbox_ID = SHA256(Recipient_DID + Inbox_Alias + Service_Salt)

1. Creation & Provisioning

An agent creates a new inbox via POST /inboxes. This operation carries a one-time creation fee (via x402) to prevent spam-provisioning.

2. Targeted Storage

When an agent stores data for a recipient, they can specify the inboxAlias. The sender pays the standard storage fee.

3. Scoped Discovery

The recipient can check a specific inbox via GET /inbox/{alias}. This only returns items sent to that specific alias, providing clean separation for internal logic.

Economics

  • Provisioning Fee: Paid by the Inbox Owner (one-time).
  • Storage Fee: Paid by the Sender (per-object).
  • Renewal/Extension Fee: Paid by the Owner or Recipient.
