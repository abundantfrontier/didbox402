The Engine: Architecture

didbox402 is built on the Cloudflare Edge. It utilizes serverless primitives to ensure that the storage is as close to the agents as possible, minimizing latency for cross-agent collaboration.

Component Stack

  • Hono: A high-performance middleware framework handling routing, DID verification, and x402 handshakes.
  • R2 (Blob Storage): Stores the ciphertext payloads. The server treats these as opaque blobs.
  • D1 (SQL Metadata): Stores storage IDs, salted DID hashes, and expiration lease data.

Data Flow: The "Box" Lifecycle

The Edge Observation: Storage is not permanent by default. It is a lease.
  1. Creation: Client posts ciphertext + x402 payment. Worker stores blob in R2 and metadata in D1.
  2. Active: The box is accessible via /retrieve/{id} to authorized DIDs.
  3. Extension: Owner can call /extend with a new payment to push back expires_at.
  4. Expiration: Once now > expires_at, the Worker refuses GET requests.
  5. Purge: A scheduled task (Janitor) deletes the R2 object and D1 row to free resources.

Atomic Constraints

  • If R2 upload fails, the D1 record is never created.
  • If the lease expires, the D1 record remains until the Janitor purges it, but retrieval is blocked at the edge.
