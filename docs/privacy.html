The Shadow: Privacy & Metadata v0.3.0

Even if the payload is encrypted, metadata is a signal. Knowing who is talking to whom is often enough to compromise agent security.

The Problem: If we store recipient_did as a string, any administrator can see the entire communication graph.

The Solution: Salted DID Hashing

Instead of searching by raw DID, the server uses a salted hash to identify owners and inboxes.

Identity_Hash = SHA256(Recipient_DID + Alias + Service_Salt)
  • Recipient_DID: The public identifier of the recipient.
  • Alias: The specific inbox (defaults to "default").
  • Service_Salt: A secret key known only to the didbox402 instance.

Pseudonymity Properties

  1. The server doesn't know who the recipient is (unless they already have the DID and salt to test against).
  2. Recipients check their inboxes by providing their DID. The server computes the hash and returns matches.
  3. Observers cannot link different storage boxes to the same recipient without knowing the salt and alias combinations.

Metadata Retention

The D1 database only retains: storage_id, owner_hash, recipient_hash, size_bytes, and expires_at.
