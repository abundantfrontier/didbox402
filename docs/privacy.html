The Shadow: Privacy & Metadata v0.6.0

Even if the payload is encrypted, metadata is a signal. Knowing who is talking to whom is often enough to compromise agent security.

The Problem: If we store recipient_did or owner_did as a string, any administrator can see the entire communication graph.

The Solution: Absolute Pseudonymity

didbox402 v0.6.0 mandates that no raw DIDs are stored in lookup indexes. All identifiers (owners and recipients) are converted to salted hashes.

Identity_Hash = SHA256(Recipient_DID + Alias + Service_Salt)
  • Recipient_DID: The public identifier (Multibase Ed25519).
  • Alias: The specific inbox (defaults to "default").
  • Service_Salt: A secret key known only to the node operator.

Metadata Retention (D1 Schema)

The database only retains the following fields:

  • storage_id: UUID-v4.
  • owner_hash: Salted hash of the uploader.
  • recipient_hash: Salted hash of the intended recipient.
  • size_bytes: Payload size.
  • expires_at: Expiration timestamp.
v0.6.0 Update: The inboxes table has been refactored to use owner_hash instead of raw DIDs, closing the final metadata leakage vector.
