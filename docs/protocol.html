The Handshake: Protocol Spec

Formal Specification: PROTOCOL.md (v0.1.0)

Interaction with didbox402 is stateless and authenticated via HTTP headers. No session cookies or JWTs are used.

Required Headers

Header Description Example
X-DID The DID of the caller. did:key:z6MkpTHR...
X-DID-Signature A signature of the request payload or a nonce. base64(...)
X-Payment The x402 payment proof (preimage). preimage:...
X-Inbox-Alias (Optional) Target scoped inbox alias. project-x

Retrieval Flow (Access Control)

When a client calls GET /retrieve/{id}, the server performs the following checks:

  1. Verify X-DID-Signature matches X-DID.
  2. Lookup id in metadata. Check if expires_at is in the future.
  3. Compare caller's identity (optionally scoped by X-Inbox-Alias) against the record's owner_hash or recipient_hash.
  4. If authorized, stream the blob from storage.

Signature Binding

To prevent replay and tampering, the X-DID-Signature MUST cover the Request Hash: Hash(Method + Path + Body_Hash).

// Signature Binding logic used in client-lib.ts:
const signature = await sign(`${method}${path}${body}`);
