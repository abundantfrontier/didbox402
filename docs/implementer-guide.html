This guide provides a comprehensive technical blueprint for teams building didbox402-compatible servers or clients. It outlines the mandatory invariants required to claim protocol compliance.

1. Introduction

didbox402 is a protocol for ephemeral, paid, and verifiable storage. It is designed for autonomous agents that need to hand off state or communicate securely without persistent accounts.

Protocol vs Product: The core protocol defines how agents authenticate, pay, and store data. Commercial products build on this by providing high-availability hosting, multi-chain payment support, or advanced audit tools.

2. Core Concepts

  • DID Authentication: Identity is proven per-request via X-DID and a signature.
  • Temporal Binding: Every request is timestamped (X-DID-Timestamp) and bound to a 5-minute validity window.
  • Dual-Rail Economics: Nodes must support at least one standard rail (L402 or x402) for lease payments.
  • Salted Privacy: All internal indexes use SHA256(DID + Alias + Secret_Salt) to ensure pseudonymity.

3. Implementing a Compatible Server

Minimum Requirements

  • Implement verifyDidSignature middleware using standard Ed25519 did:key parsing.
  • Implement Nonce Tracking to reject replayed signatures.
  • Enforce the ±5 minute drift window for timestamps.

The 402 Handshake

When a payment is required (Store/Extend), the server MUST return a 402 with the appropriate challenge headers:

WWW-Authenticate: L402 macaroon="...", invoice="..."
PAYMENT-REQUIRED: {base64_requirements}

Practical Implementation Examples

Example: Minimal Hono Server (Node.js)

This snippet demonstrates how to implement the core 402 challenge flow in a standard Node.js server using Hono.

import { Hono } from 'hono';
import { sha256 } from '@noble/hashes/sha2';

const app = new Hono();

app.post('/store', async (c) => {
  const did = c.req.header('X-DID');
  const sig = c.req.header('X-DID-Signature');
  const paymentProof = c.req.header('Authorization'); // or PAYMENT-SIGNATURE
  
  // 1. Check for payment
  if (!paymentProof) {
    // Generate standard L402 challenge
    const invoice = "lnbc100n1p..."; 
    const macaroon = Buffer.from(JSON.stringify({ amount: 100 })).toString('base64');
    
    c.header('WWW-Authenticate', `L402 macaroon="${macaroon}", invoice="${invoice}"`);
    return c.json({ error: 'Payment Required', amount_satoshis: 100 }, 402);
  }

  // 2. Verify DID Signature and Payment (Logic implemented in middleware/lib)
  // ...
  
  return c.json({ storageId: 'uuid-123' });
});

export default app;

Example: Custom Node.js Client (Manual 402 Handling)

This example shows how to interact with a didbox402 node without using the automated SDK interceptors.

import { signRequest } from '@didbox/sdk-crypto';

const MY_DID = "did:key:z6Mk...";
const MY_KEY = Buffer.from("...", "hex");
const NODE_URL = "https://node.didbox.com";

async function storeData(ciphertext) {
  const timestamp = Date.now();
  const path = "/store";
  const body = JSON.stringify({ ciphertext, durationHours: 1 });
  
  // Create temporal signature
  const sig = await signRequest(MY_KEY, "POST", path, body, timestamp);

  // Initial Attempt
  let res = await fetch(`${NODE_URL}${path}`, {
    method: "POST",
    headers: {
      "X-DID": MY_DID,
      "X-DID-Signature": sig,
      "X-DID-Timestamp": timestamp.toString(),
      "Content-Type": "application/json"
    },
    body
  });

  if (res.status === 402) {
    const challenge = res.headers.get("WWW-Authenticate");
    const { invoice, macaroon } = parseL402(challenge);
    
    // Settle via Lightning Wallet (Alby/WebLN)
    const { preimage } = await window.webln.sendPayment(invoice);
    
    // Retry with proof
    res = await fetch(`${NODE_URL}${path}`, {
      method: "POST",
      headers: {
        "X-DID": MY_DID,
        "X-DID-Signature": sig, // Reuse same sig within window
        "X-DID-Timestamp": timestamp.toString(),
        "Authorization": `L402 ${macaroon}:${preimage}`,
        "Content-Type": "application/json"
      },
      body
    });
  }

  return res.json();
}

9. Migrating Data Between Nodes (v0.7.0+)

didbox402 supports Sovereign Mobility — the ability for clients to move their data from one node to another in a verifiable way.

Core Concepts

  • A Migration Proof (signed MigrationAuthorization) is obtained from the source node.
  • The client is responsible for downloading the data and re-uploading it to the destination.
  • In Phase 1, the destination node does not yet act on the Migration Proof (this comes in later phases).

Basic Migration using the SDK

import { DidBoxClient } from '@didbox/sdk-core';
import { signRequest } from '@didbox/sdk-crypto';

const client = new DidBoxClient({
  baseUrl: "https://source-node.didbox.com",
  did: MY_DID,
  signRequest: (ts, method, path, body) => signRequest(MY_KEY, method, path, body, ts)
});

// Option 1: Get a Migration Proof (recommended low-level approach)
const { authorization, verified } = await client.getMigrationProof("storage-abc123");

if (!verified) {
  console.error("Failed to verify migration proof");
  return;
}

console.log("Migration Proof obtained. Remaining lease:", authorization.remaining_lease_hours, "hours");

// Option 2: Use the basic migrate() helper (convenience)
const result = await client.migrate("storage-abc123", {
  destinationUrl: "https://dest-node.didbox.com",
  newDurationHours: 72
});

console.log("Migration complete. New storage ID:", result.newStorageId);

Using a Client for a Different Node

If you need to obtain a Migration Proof from a node you are not primarily connected to: 

const sourceClient = DidBoxClient.forNode("https://old-node.didbox.com", {
  did: MY_DID,
  signRequest: mySignFunction
});

const proof = await sourceClient.getMigrationProof(storageId);

Important Notes (Phase 1)

  • The migrate() helper is intentionally basic. It does not yet present the Migration Proof to the destination node.
  • You are responsible for paying any egress fees on the source node.
  • Both source and destination nodes must be running v0.7.0+ for full functionality.

Client SDK Guidance and Recommendations

didbox402 is intentionally a client-heavy protocol. The server is deliberately kept simple and stateless so that providers remain commodity infrastructure. Most intelligence, resilience, and ergonomics therefore live in client libraries and SDKs.

The normative (mandatory) client behaviors are defined in PROTOCOL.md §9.1. The following section describes what high-quality client SDKs SHOULD provide on top of the core protocol. These recommendations are not required for basic protocol compliance, but they are necessary for a usable agent and developer experience.

Recommended High-Level Abstractions

  • Automatic payment handling — A DidBoxClient (or equivalent) that can detect 402 responses, negotiate payment via pluggable L402/x402 providers, and transparently retry the original request.
  • Migration orchestration — Helpers such as getMigrationProof(), migrate(), and DidBoxClient.forNode() that make Sovereign Mobility straightforward for clients.
  • Lease lifecycle support — Cost estimation using /price, expiry monitoring, and assisted lease extension.
  • Resilience patterns — Built-in support for client-side mirroring of important conversations/threads into per-DID archive inboxes, plus reconciliation helpers (see the Deletion Semantics design document).
  • Group communication helpers — Ergonomic APIs for the per-DID signing + shared symmetric encryption pattern without forcing application code to manage the details.
  • Paid content distribution support — Helpers for creators and platforms using the off-protocol payment model (see the Paid Content Distribution design pattern), including lease management, key handling coordination, and private delivery flows.
  • Discovery and configuration — Automatic fetching and caching of /.well-known/didbox-configuration, with exposure of supported rails, limits, and node identity.
  • Clock and error recovery — Automatic use of server Date headers for drift correction and clear error types for common failure modes (drift, replay, payment required, etc.).

Reference Packages

The official packages aim to implement the above:

  • @didbox/sdk-crypto — Low-level Ed25519 signing, request binding, migration proof verification.
  • @didbox/sdk-payments — Rail negotiation and proof handling.
  • @didbox/sdk-core — High-level DidBoxClient that combines the above with convenient methods.

Application authors and agent frameworks are strongly encouraged to build on these packages rather than re-implementing protocol details.

10. Server Implementation Requirements (v0.7.0+)

Supporting Sovereign Mobility

To be compliant with v0.7.0, your node MUST implement POST /migrate/{id}/authorize.

Core Requirements

  • Require valid DID authentication (the caller must be the owner of the storage object).
  • Return 404 for unknown storage IDs, 403 for non-owners, 410 for expired leases.
  • Compute ciphertext_hash (sha256:) of the current blob.
  • Calculate remaining_lease_hours as max(1, ceil(remaining time in hours)).
  • Sign the authorization using JCS canonicalization (RFC 8785) + your node's dedicated Ed25519 key.
  • Publish your node identity in /.well-known/didbox-configuration (see below).

Node Identity Configuration (Mandatory)

Every production node in v0.7.0 MUST be configured with a dedicated Ed25519 keypair for signing Migration Authorizations.

# wrangler.toml or environment
NODE_DID="did:key:z6MkYourNodeIdentity..."
NODE_SIGNING_PRIVATE_KEY="32-byte hex private key (64 hex chars)"

The corresponding public key will be automatically derived and published in discovery.

Important Environment Variables (v0.7.0)

Variable Required? Purpose
SERVICE_SALT Yes (prod) High-entropy secret for DID hashing (≥128 bits)
NODE_DID Yes (v0.7.0+) Node's did:key identity (Ed25519)

11. Guidance for Commercial & Fleet Operators (v0.7.0+)

Partial Rail Support

Commercial nodes may advertise only a subset of rails (e.g., ["x402"]). This is fully compliant. Ensure your discovery response and conformance testing scope reflect only the rails you actually support.

Node Identity for Multi-Node Fleets

For high-availability or multi-region deployments, you have two compliant options:

  • Shared signing key: Use the same Ed25519 key across all nodes in your fleet. This allows any node to issue verifiable Migration Proofs for leases stored anywhere in the fleet.
  • Per-node keys: Each node uses its own identity. Clients will rely on the source_node field in Migration Proofs.

Shared keys are often simpler for Phase 1 Sovereign Mobility.

SERVICE_SALT Considerations

The SERVICE_SALT is effectively immutable. Rotating it will break all existing inbox and lease lookups. Treat it as a long-lived secret with the same protection level as your node signing key. Future protocol versions may introduce rotation support.

Admin / Operational Endpoints

It is acceptable (and recommended for production) to expose strongly authenticated admin endpoints (e.g., POST /admin/purge protected by DID ACL or a strong shared secret) for emergency or on-demand operations. These endpoints are outside the public protocol.

Pricing Flexibility

You may implement per-DID or tiered pricing. As long as the authenticated GET /price returns the rates that will be used for that specific caller, and those rates are used to compute the 402 challenge, the implementation remains compliant.

x402 (Base USDC) Implementation Notes

  • Use the official Base USDC contract: 0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913 (mainnet)
  • For testnet, use the Base Sepolia USDC contract.
  • Require at least 1 confirmation for production.
  • Store used txHash values in used_payments with appropriate expiry.

Group Communication (Exploratory / Future Direction)

Note: This is an exploratory client-side pattern, intended for post-v0.7.0 use cases. It does not require any server-side changes.

For secure group chat or shared workspaces across multiple DIDs while keeping the server blind, see the design document:

Group Communication Design on didbox402

This approach uses individual DIDs for signing (provenance) + a shared symmetric group key for encryption. No server-side changes are required.

Client-Side Resilience Patterns (Deletion and Expiry)

Because the protocol treats deletion as a best-effort operational obligation rather than a cryptographically provable event, clients are responsible for managing resilience against lease expiry or provider non-compliance. A recommended pattern is client-side mirroring of important threads or conversations into per-DID archive inboxes. This is documented in:

Deletion Semantics, Attestations, and Client-Side Resilience

This pattern requires no changes to the server or protocol and is especially relevant for group communication and long-lived agent handoffs.

Paid Content Distribution (Off-Protocol Payments)

Many legitimate commercial use cases involve creators or publishers monetizing digital content while preserving privacy for both buyer and seller. The recommended approach keeps the actual payment outside of didbox402 (direct between the parties) and uses the protocol only for private delivery of the encrypted content. This model avoids turning storage nodes into financial intermediaries and aligns with the protocol’s core privacy principles.

For details and recommended patterns, see:

Privacy-Preserving Paid Content Distribution

Example use cases include independent journalism and research, professional knowledge products, research datasets, and creative assets.

Background Purge and Janitor Strategies at Scale

The reference implementation exposes GET /janitor/purge (protected by X-Admin-Token) as a simple manual trigger. It processes expired records in small batches (default 50) and also cleans old nonces.

For production deployments handling thousands or millions of leases, you should move to automated background purging. Recommended approaches include:

  • Cloudflare Durable Object alarms (per-shard or per-tenant) for scheduled, rate-limited purge runs directly in the Worker environment.
  • Cloudflare Workflows or Cron Triggers for centralized scheduling with backpressure and observability.
  • External scheduler (e.g., a small worker or GitHub Action + Wrangler) that calls your admin purge endpoint on a schedule.
  • Incremental + bounded deletes — always use LIMIT and process in small batches to avoid long-running transactions or D1 timeouts.

Whichever mechanism you choose, ensure it runs at least hourly (finer granularity is better for very short leases) and is observable (logs, metrics on purged count and duration).

Replay Protection Store Lifecycle (used_payments and nonces)

Both nonces (DID signature replay) and used_payments (L402/x402 payment proof replay) tables must be cleaned to prevent unbounded growth.

  • nonces are short-lived (tied to the 5-minute drift window + small grace). The reference janitor already deletes them.
  • used_payments rows live for the duration of the lease they paid for (plus a grace period). This can be up to max_lease_hours (8760 hours / 1 year) for long leases.

Operational considerations:

  • Under high churn (many 1-hour leases) you will accumulate many short-lived rows.
  • Under long leases you accumulate fewer rows but they stay for a very long time.
  • In any horizontally scaled fleet, the replay check must be strongly consistent across nodes that can serve the same request.
  • The current reference implementation inserts rows with an expires_at column and an index, but does not automatically delete old used_payments rows. You must add cleanup (e.g., DELETE FROM used_payments WHERE expires_at < ?) to your janitor or a separate scheduled task.

Monitor table size and cleanup lag in production. D1 has practical limits; very high-volume nodes may eventually need sharding or a separate ledger for payment proofs.

Node Signing Key Management

The Ed25519 key published in node_identity (used to sign Migration Authorizations) should be treated with high security — equivalent to a payment provider secret.

  • Store it in an appropriate secret manager (not in source control or plain environment variables in production).
  • Rotation is allowed: publish a new node_identity in discovery. Previously issued proofs remain valid until their natural expires_at.
  • For fleets: sharing one key across nodes simplifies Migration Proof verification for clients. Using per-node keys is also compliant but requires clients to track source_node.
  • Never use the same key for both node identity / migration proofs and day-to-day admin operations if you can avoid it.

Common Errors & How to Fix Them

Error Code Likely Cause Resolution
401: Timestamp window Client clock drift > 5 mins. Sync client clock with NTP or use server time from a previous response.
401: Invalid Signature Signature does not cover full request hash. Verify Method, Path, and BodyHash are included in order.
401: Replay detected Same signature used twice. Generate a fresh timestamp and signature for every new request.
413: Payload too large Sent > 10MB ciphertext. Chunk large data or use a node with higher limits.

8. Conformance & Testing

Use the @didbox/conformance suite to validate your implementation. Your node must pass the following scenarios:

  • Auth: Replay rejection, drift window enforcement, malformed DID rejection.
  • Privacy: Proof of salted hashing (no raw DIDs in DB).
  • Economics: Correct 402 response code and amount calculation.
