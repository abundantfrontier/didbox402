Implementer's Guide v0.6.0

This guide provides technical instructions for developers building a compliant didbox402 node. By following these requirements, your implementation will be interoperable with the official SDKs and other compliant clients.

1. The Verification Gauntlet

Any compliant node MUST implement the following five pillars of security and economics:

  • L402/x402 Verification: Real preimage or signature validation (no mocks).
  • Ed25519 Auth: Standard did:key multibase parsing and signature verification.
  • Replay Protection: Mandatory nonce tracking and ±5m drift window.
  • Salted Privacy: All ID indexes MUST be hashed using a secret node salt.
  • Lease Lifecycle: Automated purging of expired blobs and metadata.

2. Choosing Your Infrastructure

While the reference implementation uses Cloudflare, the protocol is Infrastructure Agnostic. You can implement didbox402 on:

  • AWS: Lambda (Compute) + S3 (BLOB) + RDS/Dynamo (Metadata).
  • GCP: Functions (Compute) + GCS (BLOB) + Cloud SQL (Metadata).
  • Self-Hosted: Node.js/Bun (Compute) + MinIO (BLOB) + SQLite/Postgres (Metadata).

3. Implementing the Handshake

The signature verification is the most critical component. Ensure your middleware handles the body buffer correctly for Signature Binding.

// Example Middleware Logic:
const body = await req.text();
const bodyHash = sha256(body);
const requestHash = sha256(`${timestamp}${method}${path}${bodyHash}`);
const isValid = ed25519.verify(signature, requestHash, publicKey);

4. Handling Payment Challenges

Your node MUST respond with a 402 Payment Required when a lease is requested or extended without a valid proof.

// x402 Challenge Header:

5. Capability Advertising

Nodes SHOULD implement a GET /price or GET /info endpoint so clients can discover supported rails and current rates autonomously.
