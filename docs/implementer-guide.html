Implementer's Guide v0.6.0

This guide provides a comprehensive technical blueprint for teams building didbox402-compatible servers or clients. It outlines the mandatory invariants required to claim protocol compliance.

1. Introduction

didbox402 is a protocol for ephemeral, paid, and verifiable storage. It is designed for autonomous agents that need to hand off state or communicate securely without persistent accounts.

Protocol vs Product: The core protocol defines how agents authenticate, pay, and store data. Commercial products build on this by providing high-availability hosting, multi-chain payment support, or advanced audit tools.

2. Core Concepts

  • DID Authentication: Identity is proven per-request via X-DID and a signature.
  • Temporal Binding: Every request is timestamped (X-DID-Timestamp) and bound to a 5-minute validity window.
  • Dual-Rail Economics: Nodes must support at least one standard rail (L402 or x402) for lease payments.
  • Salted Privacy: All internal indexes use SHA256(DID + Alias + Secret_Salt) to ensure pseudonymity.

3. Implementing a Compatible Server

Minimum Requirements

  • Implement verifyDidSignature middleware using standard Ed25519 did:key parsing.
  • Implement Nonce Tracking to reject replayed signatures.
  • Enforce the ±5 minute drift window for timestamps.

The 402 Handshake

When a payment is required (Store/Extend), the server MUST return a 402 with the appropriate challenge headers:

WWW-Authenticate: L402 macaroon="...", invoice="..."
PAYMENT-REQUIRED: {base64_requirements}

Storage Lifecycle

  • Atomic Store: Data in R2/S3 must be consistent with metadata in D1/SQL.
  • Expiration: Retrieval MUST be blocked as soon as now > expires_at.
  • Physical Purge: A background process must physically delete blobs to prevent resource exhaustion.

4. Implementing a Client

Key Management

Clients are responsible for their own Ed25519 keys. The protocol expects standard did:key identifiers.

Handling 402 Challenges

A compliant client SHOULD:

  1. Attempt a request without payment headers.
  2. On 402, parse the WWW-Authenticate or PAYMENT-REQUIRED headers.
  3. Settle the payment via a connected wallet (WebLN or EIP-1193).
  4. Retry the request with Authorization or PAYMENT-SIGNATURE.

5. Extension Points

Commercial providers can extend didbox402 at these points:

  • Payment Rails: Add support for Solana, Lightning Service Providers (LSPs), or custom credit systems.
  • Storage Backends: Move from Cloudflare R2 to IPFS, Arweave, or enterprise-grade S3 clusters.
  • Authorization: Implement Verifiable Credentials (VCs) for gated storage tiers.

Practical Implementation Examples

Example: Minimal Hono Server (Node.js)

This snippet demonstrates how to implement the core 402 challenge flow in a standard Node.js server using Hono.

import { Hono } from 'hono';

const app = new Hono();

app.post('/store', async (c) => {
  const paymentProof = c.req.header('Authorization'); // or PAYMENT-SIGNATURE
  
  if (!paymentProof) {
    // Issue L402 Challenge
    c.header('WWW-Authenticate', 'L402 macaroon="...", invoice="lnbc..."');
    return c.json({ error: 'Payment Required', amount_satoshis: 100 }, 402);
  }

  // 1. Verify DID Signature (Shield)
  // 2. Verify Payment Proof (Rail)
  // 3. Store Data (Engine)
  
  return c.json({ storageId: 'uuid-123' });
});

export default app;

Example: Manual Client-Side 402 Handling

If you choose not to use the Intelligent Interceptor, you can handle the economic handshake manually as follows:

try {
  const res = await client.store(payload, 24);
} catch (err) {
  if (err.status === 402) {
    const { invoice } = err.challenge;
    console.log(`Payment required. Paying invoice: ${invoice}`);
    
    // 1. User/Agent pays the invoice
    const { preimage } = await myWallet.pay(invoice);
    
    // 2. Retry with the proof
    const finalRes = await client.store(payload, 24, { payment: preimage });
    console.log('Successfully stored after payment:', finalRes.storageId);
  }
}

7. Running Outside Cloudflare

To ensure true protocol neutrality, didbox402 can be deployed on standard cloud providers or bare metal.

Storage Alternatives

  • BLOB: Use the S3-compatible API (supported by AWS S3, GCS, and MinIO).
  • Metadata: Swap D1 for PostgreSQL or MySQL.

Compute Portability

The Hono framework runs on any Node.js, Deno, or Bun runtime. You can deploy as a Docker container or a serverless function (AWS Lambda / Google Cloud Functions).

8. Conformance & Testing

Use the @didbox/conformance suite to validate your implementation. Your node must pass the following scenarios:

  • Auth: Replay rejection, drift window enforcement, malformed DID rejection.
  • Privacy: Proof of salted hashing (no raw DIDs in DB).
  • Economics: Correct 402 response code and amount calculation.

7. Security Considerations

  • Salt Management: The node salt must be kept secret. Rotating the salt will invalidate all existing inbox discoverability.
  • Signature Binding: Always verify that the signature covers the specific Method + Path + Body being requested.
