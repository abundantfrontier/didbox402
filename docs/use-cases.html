Practical Use Cases v0.6.2

This document describes real-world usage patterns built on top of the didbox402 protocol. These patterns demonstrate how the core primitives of ephemeral, paid, and verifiable storage can be applied in practice.

Single-Use DID Handoff

The Single-Use DID Handoff pattern allows two parties (humans or agents) to securely exchange their long-term DIDs and public keys using a temporary, short-lived didbox inbox.

The Problem: Bootstrapping secure communication over insecure channels (like email) without exposing permanent identities early.

The Pattern

  1. Create Temporary Inbox: Alice generates a temporary DID and creates a short-lived didbox inbox (e.g., 1-24 hour lease).
  2. Share Location: Alice sends the temporary address to Bob via email or another channel.
  3. Identity Deposit: Bob writes his real long-term DID and public key into that temporary inbox.
  4. Retrieval: Alice reads Bob's real DID and begins secure communication in their permanent, long-term channels.
  5. Self-Destruct: The handoff inbox automatically expires and is purged by the Janitor.

Human-to-Human Exchange

Provides a simple, relatively secure way to bootstrap encrypted communication:

  • Share links via QR codes or messaging apps.
  • Clients handle the background DID generation and encryption.
  • Minimizes exposure of long-term identities over insecure channels.

Agent-to-Agent Exchange

Enables autonomous systems to establish secure, long-term relationships without human intervention:

  • Agents programmatically discover handoff locations.
  • Support for capability-based access control (e.g., one-time write tokens).
  • Mutual handoffs (both parties exchanging temporary inboxes simultaneously).

Benefits of This Pattern

  • Reduces exposure of long-term identity over insecure links.
  • Leverages existing protocol features (Expiration, L402 Macaroons).
  • Naturally limits the attack window via ephemerality.

Implementation Notes

This pattern requires NO changes to the core protocol. It utilizes:

  • Short-lived inboxes via expires_at.
  • Client-side E2EE for the handoff payload.
  • Capability discovery via /.well-known/didbox-configuration.

Secure Results Drop-off (Human-to-Agent)

In this pattern, a human leases a "Secure Drop-off Box" for an autonomous agent to deliver results once a long-running task is complete.

The Problem: A human wants an agent to perform a task and deliver a potentially large or sensitive report without maintaining a permanent server or shared account.

The Pattern

  1. Human Provisions Box: The human creates a didbox lease for 48 hours, specifying the agent's DID as the recipientDid.
  2. Link Sharing: The human provides the storageId (and potentially the node URL) to the agent as part of the task instructions.
  3. Agent Delivers: Once the task is finished, the agent encrypts the result and performs a POST /extend (if needed) or simply writes to an agreed-upon inbox alias.
  4. Human Retrieves: The human retrieves the report at their convenience.

Benefits

  • Zero-Account Interaction: No need for the agent to have an AWS/Dropbox account.
  • Pre-paid Convenience: The human (the employer) pays the storage fee upfront.
  • Automatic Cleanup: Sensitive reports vanish once the lease expires.

Verified Data Request (Agent-to-Human)

An autonomous agent creates a paid "Inbound Box" and requests a specific piece of data from a human user.

The Problem: An agent needs a human to provide a specific file (e.g., a signed PDF or KYC document) but wants to make the process frictionless for the human.

The Pattern

  1. Agent Creates Inbox: The agent calls POST /inboxes with a custom alias (e.g., "contract-upload") and pays the provisioning fee.
  2. Request Sent: The agent sends a link to the human (via email or frontend) pointing to this paid inbox.
  3. Human Uploads: The human uses a simple web client to upload the file. Since the agent already provisioned the inbox, the human's UX is frictionless.
  4. Agent Processes: The agent monitors the inbox, retrieves the data, and proceeds with its workflow.

Benefits

  • Agent-Led Economics: The agent manages its own budget to facilitate human contributions.
  • Scoped Privacy: The human only sees and interacts with the specific inbox alias intended for them.
  • Stateless Collection: The agent doesn't need to host a complex file-upload server.

Experimental / Future Patterns

These patterns describe capabilities enabled by the protocol's extensible 402-based architecture but may not be implemented in all nodes.

High-Speed Coordination (Mem-Cache)

Agents use a specialized memory-backed storage class for sub-second coordination.

The Problem: Standard R2/S3-backed storage has too much latency for agents performing rapid negotiation or lock-stepping.
  • Mechanism: Client requests storageClass: "mem-cache".
  • Performance: Node uses RAM-only storage (e.g., Redis or Durable Objects).
  • Economics: High base rate, very short TTLs (minutes).

Autonomous Compute Drop-off

A node performs a deterministic task and delivers the result to a DID-scoped inbox.

The Problem: An agent needs to perform a heavy computation (like generating a ZK proof or resizing an image) but wants to keep the result in the same DID-verified ecosystem.
  • Mechanism: Client pays a single L402 invoice covering both the "Compute Gas" and the "Result Storage."
  • Delivery: The node acts as a "Virtual Agent," depositing the result into the requester's inbox.
  • Verifiability: The result is signed by the node's DID, ensuring provenance.
