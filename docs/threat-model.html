The Shield: Threat Model v0.9.1 (Normative Reference)

Honesty starts with admitting what didbox402 cannot protect. While we aim for maximum privacy, the physical reality of hosting and networking creates metadata "exhaust."

1. Key Assets & Trust Boundaries

Asset Trust Boundary Protection
Plaintext Data Client-Side End-to-End Encryption (E2EE)
Identity (DID) Protocol-Wide Salted Hashing & Ed25519 Signatures
Funds (Sats/USDC) Wallet-Side 402 Challenge-Response Handshake (micropayment nodes)
Org Entitlement (API Keys) Operator / Enterprise Perimeter SHA-256 hashed secrets, private network deployment, rotation

2. Main Threat Categories

Authentication (The Shield)

Threat: An attacker intercepts a signed storage request and resubmits it to drain funds or overwrite data.

Mitigation: Mandatory Nonce Tracking and Temporal Request Hash binding. Signatures are single-use and valid for ±5 minutes.

Payments (The Rail)

Threat: A provider accepts payment but refuses retrieval or deletes data early.

Mitigation: didbox402 is a Commodity Protocol. Agents should diversify storage across multiple nodes. Provider reputation acts as a market signal.

Privacy (The Shadow)

Threat: The storage provider observes who is communicating with whom by observing DID hashes.

Mitigation: Salted DID Hashing for all lookup indexes. Observers cannot link storage boxes to recipients without knowing the secret node salt.

Availability (Infrastructure)

Threat: Malicious actors trigger the Janitor endpoint to disrupt service.

Mitigation: Mandatory X-Admin-Token requirement for all Janitor operations.

Entitlement Billing (v0.9.0+)

Threat: Exfiltration of X-DIDBOX-Entitlement API keys allows any party to pair the key with an arbitrary agent DID and consume org storage quota. Keys are org-wide bearer credentials in Phase 1 (not bound per DID).

Mitigation: Deploy entitlement nodes on private networks; use ≥128-bit CSPRNG secrets; rotate keys with overlap windows; never log entitlement headers; replicate hash config consistently across HA replicas. Phase 2 capability tokens with per-DID sub claims reduce blast radius.

Dual Billing Mode Confusion

Threat: Clients mis-handle nodes after billing_mode changes, or attach entitlement headers to public micropayment nodes expecting free access.

Mitigation: Clients MUST read discovery per URL; micropayment nodes ignore entitlement headers and still require 402 settlement. Entitlement nodes MUST NOT return 402 for billable operations.

Ephemerality and Deletion

Threat: A provider fails to purge expired leases or retains ciphertext beyond the agreed lease period.

Mitigation: Deletion is a best-effort operational obligation on the provider (see PROTOCOL.md §1.2 and §10.2). Clients can reduce exposure through Sovereign Mobility and client-side redundancy patterns such as conversation mirroring. See the design document Deletion Semantics, Attestations, and Client-Side Resilience for details on remaining trust boundaries and practical mitigation approaches. No cryptographic proof of deletion is currently defined.

3. Trust Assumptions

  • Client Integrity: We assume the client handles their Ed25519 keys securely and never shares the plaintext.
  • Infrastructure Bit-Integrity: We trust the underlying cloud (R2/S3) to maintain data for the lease duration.
