The Shield: Threat Model

Honesty starts with admitting what didbox402 cannot protect. While we aim for maximum privacy, the physical reality of hosting and networking creates metadata "exhaust" that users must understand.

1. Metadata Leakage (The Social Graph)

Threat: The storage provider can see who is communicating with whom by observing IP addresses and DID hashes.

Mitigation: Use Salted DID Hashing for inbox IDs and encourage agents to use Tor/VPNs to mask network-level origin.

2. Payload Size & Frequency (Traffic Analysis)

Threat: Even if data is encrypted, the size of the ciphertext and the frequency of retrieval can leak the type of data (e.g., streaming video vs. small text state).

Mitigation: Clients should implement Padding (making all blobs a fixed size or multiples of a block) to normalize the traffic profile.

3. Malicious Provider (The Data Trap)

Threat: A provider could accept payment but delete data early, or refuse retrieval despite a valid lease.

Mitigation: didbox402 is a Commodity Protocol. Agents should diversify storage across multiple nodes. The reputation of a provider (e.g. Omnibond) becomes a market signal for reliability.

4. Replay Attacks

Threat: An attacker intercepts a signed storage request and resubmits it to drain the client's funds or overwrite data.

Mitigation: X-DID-Signature must bind to the Request Hash (Method + Path + Body Hash) and should include a timestamp or nonce to ensure one-time validity.

Trust Assumptions

  • Client Integrity: We assume the client handles their private keys securely and encrypts data before sending it.
  • Infrastructure Durability: We trust the underlying cloud provider (R2/S3) to maintain the physical bit-integrity for the duration of the lease.
