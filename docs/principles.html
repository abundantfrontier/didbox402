The Foundation: Storage Principles v0.5.0

To achieve hands-off scaling and LLM compatibility, didbox402 follows a strict separation of concerns between BLOB storage (Files) and Relational storage (Database).

Scaling Philosophy: Assume the protocol will handle billions of small objects. No single human should ever need to "rebalance" the cluster.

1. Payloads are Opaque (BLOBs)

Constraint: The core server never parses the content of a stored item.

Why: Scaling file storage (R2/S3) is a solved problem. By keeping payloads opaque and client-side encrypted, we offload the heavy lifting to specialized infrastructure that scales horizontally without database locking overhead.

2. Metadata is Structured (DB)

Constraint: Only use the database for identity, expiration, and discovery (hashes).

Why: Relational databases are for searching, not storing. By keeping the DB rows tiny (Fixed-size hashes and timestamps), we ensure that the index stays in-memory (hot) for millions of records, enabling low-latency "Inbox" lookups.

3. LLM-Friendly Transparency

Constraint: Protocol responses must be self-describing JSON with explicit units (e.g., size_bytes, not size).

Why: Agents need to reason about their own resource usage. If an LLM-agent can't understand the pricing response or the metadata headers, it can't make autonomous decisions about whether to extend a storage lease.

4. Automatic Garbage Collection

Constraint: Storage is a lease, not a right.

Why: Hands-off scaling requires that dead data removes itself. By tying every BLOB to a mandatory expires_at timestamp in the DB, the system remains lean. If a payment isn't renewed, the data is purged, preventing "abandoned-data bloat."

5. Cryptographic Sovereignty

Constraint: The provider is a "Ghost." It holds no keys and makes no claims to the data.

Why: Agents must retain full control. By using DID-based signatures and client-side encryption, the storage provider becomes a pure commodity. If a provider becomes malicious or expensive, the agent can migrate to a new node using their same DID identity.

6. Verifiable Economics

Constraint: Pricing must be deterministic and calculable offline.

Why: Agents should never be surprised by a 402 rejection. By making the pricing formula public and simple, agents can verify they are paying the fair rate and can calculate their "burn rate" for long-term storage without querying the server.

7. Stateless Autonomy

Constraint: Every request must be self-contained (Stateless).

Why: Scaling to billions of agents requires that the server doesn't "remember" the user between requests. No sessions, no cookies. Identity is proven per-request via X-DID-Signature. This allows for massive horizontal scaling across any cloud provider.

Decision Matrix

Data Type Storage Method Reasoning
Ciphertext BLOB (R2/S3) High volume, low searchability.
Expiration DB (D1/SQLite) Fast range queries for Janitor.
Inbox IDs DB (D1/SQLite) Point lookups for discovery.
