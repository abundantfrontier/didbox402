The Economics: Dynamic Pricing

Storage on the edge is a finite resource. didbox402 uses a predictable pricing model to ensure the service pays for its own infrastructure while remaining accessible.

Pricing Formula

The cost is calculated based on volume (bytes) and duration (hours).

Total Cost = (Size in MB * Hours) * Base_Rate
  • Base_Rate: Configured in wrangler.toml (e.g., 100 Satoshis per MB-Hour).
  • Currency: Satoshis (via the Lightning Network).
  • Minimum Duration: 1 Hour.
  • Maximum Duration: 8,760 Hours (1 Year).
The x402 Rail: For the MVP, we use the Lightning Network. The X-Payment header contains an LSAT (Lightning Service Authentication Token) or a simple preimage proof of a settled invoice.

Payment Specification

didbox402 nodes act as Lightning Service Providers. The payment flow follows the 402 Payment Required standard:

  1. Discovery: Client sends a request without a payment header.
  2. Challenge: Server responds with 402 Payment Required and an X-Invoice header containing a Lightning Invoice (BOLT11).
  3. Settlement: Client pays the invoice via their Lightning node/wallet.
  4. Proof: Client retries the request with X-Payment: {preimage}.
  5. Fulfillment: Server verifies the preimage and processes the request.

The "Edge" of the Handshake

The paymentMiddleware needs to know the price *before* it allows the request through. Since didbox402 price is dynamic based on the body, we follow this sequence:

  1. Client calculates local price based on their payload size and desired duration.
  2. Client generates an x402 payment for that amount.
  3. Client sends POST /store with the payment in the header and the data in the body.
  4. Server-Side:
    • Middleware reads the body to get sizeBytes and durationHours.
    • Middleware recalculates the Expected_Price.
    • If Payment_Value < Expected_Price, the request is rejected with 402 Payment Required.

Egress-Aware Pricing

On infrastructure that charges for bandwidth (e.g., AWS S3), the protocol supports a Pay-to-Download model. The GET /retrieve/{id} route requires an x402 payment matching the object's size multiplied by the provider's egress rate.
